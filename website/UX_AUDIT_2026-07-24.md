# Nexora main-site UX audit and implementation record

Date: 2026-07-24
Branch: `agent/main-site-ux-motion-polish`
Scope: `website/` public main page only. `advanced-website/` and `/advanced/` are excluded.

## Safe implementation assumptions

- Visual intensity: **medium**. Effects must be visible as polish, not as a demo scene.
- WebGL is not introduced. Existing Canvas 2D Aether and CSS gradients remain the rendering baseline.
- Safari/iOS compatibility, keyboard operation, reduced motion, touch targets and 320 px width are release-blocking requirements.
- Minimal runtime attributes and ARIA updates are allowed. Product copy, section order and information architecture remain unchanged.
- Existing live GitHub data, release selection, RU/EN, Aether, reveal, parallax and tilt implementations remain authoritative and are hardened rather than duplicated.

## Preserved strengths

- Clear dark Nexora identity with violet accents.
- Existing first-screen product mock-up and Aether background.
- Server-authority and Pulse-isolation wording in the architecture section.
- Live GitHub statistics and honest asset fallback behavior.
- Existing mobile, low-power and `prefers-reduced-motion` branches.
- Static deployment without external runtime scripts, styles or analytics.

## Confirmed UX and implementation issues

| ID | Observation | Implemented correction |
|---|---|---|
| AUD-01 | Motion durations and easing values are distributed across components. | Added centralized motion, easing, distance and glow tokens in `site-polish.css`. |
| AUD-02 | Hover, press and focus feedback is inconsistent across navigation, CTA, downloads and cards. | Added consistent hover/focus/pressed/touch states and directional icon motion. |
| AUD-03 | Mobile menu lacks Escape, outside-click, focus restoration and explicit hidden state. | Added keyboard lifecycle, focus loop, outside dismissal and responsive `aria-hidden`. |
| AUD-04 | Active navigation is visual only. | Synced the existing `.active` state to `aria-current="page"`. |
| AUD-05 | Hero reveal is section-level and lacks coordinated internal sequence. | Added a sub-one-second status/title/lead/actions/facts choreography without disabling pointer input. |
| AUD-06 | Ambient animations continue when their sections are offscreen. | Added IntersectionObserver-driven `data-ux-offscreen` and animation pause rules. |
| AUD-07 | Download cards do not expose a reusable loading/available/unavailable state to CSS and assistive technology. | Added inferred asset state, `aria-busy` and polite live regions. |
| AUD-08 | Background depth is mainly a fixed grid and Canvas layer. | Added a low-opacity CSS light/noise field with low-power and reduced-motion degradation. |
| AUD-09 | Internal diagrams are mostly static while parent cards tilt. | Added restrained hover/focus micro-motion to meters, policy bars and data-flow lines. |
| AUD-10 | Touch feedback depends primarily on CSS hover/active behavior. | Added delegated pressed-state feedback with cleanup on pointer cancellation and blur. |
| AUD-11 | Release selector waits for repository, package, releases, Actions and commit-stat requests as one batch. A single request without a timeout can leave the selector and cards in the initial loading state indefinitely. | Added a bounded GitHub request layer, cached successful release metadata and a generated `CHANGELOG.md` fallback catalog that is merged with live releases. |

## Release selector root cause and correction

The core runtime uses one `Promise.allSettled` batch for five independent GitHub data sources. `Promise.allSettled` cannot complete while any request remains pending, and the original requests had no abort deadline. The release selector was therefore not populated even when only the Actions or commit-activity endpoint stalled.

The correction does not replace the existing release renderer:

1. `release-fallback.js` provides version/date/tag metadata generated from the canonical `CHANGELOG.md`.
2. `network-resilience.js` runs before `app.js`, applies a five-second deadline only to the existing GitHub API/raw requests, and returns a standards-compatible JSON `Response` for the release endpoint when GitHub is unavailable.
3. Successful live releases are sanitized, cached and merged by `tag_name` with the fallback catalog, so real assets remain authoritative while historical versions remain visible.
4. The existing `app.js` continues to own selector options, selected-release state, asset matching and download links.

## Architecture of the change

- `release-fallback.js`: generated, non-executable release metadata derived from `CHANGELOG.md` and exposed to the composed browser runtime.
- `generate-release-fallback.mjs`: deterministic generator and `--check` freshness gate.
- `network-resilience.js`: timeout, cache, live/fallback merge and response adapter for existing GitHub requests.
- `site-polish.css`: motion tokens, component states, hero choreography, CSS light field, offscreen pause rules, responsive and accessibility fallbacks.
- `site-polish.js`: enhancement-only lifecycle for menu accessibility, active navigation semantics, download state semantics, offscreen animation suspension and touch feedback.
- `.github/workflows/pages.yml`: syntax validation, regression test and deterministic JavaScript composition before the existing application runtime.
- `validate-restored.mjs`: guards for release resilience, motion tokens, accessibility behavior, low-power/reduced-motion handling and no WebGL dependency.
- `build.json`: deploy marker for published smoke verification.

## Security and privacy review

- No external dependency, CDN, analytics, tracker, cookie or new network endpoint is added.
- The resilience layer only wraps the requests already issued by `app.js` to `api.github.com` and `raw.githubusercontent.com`.
- Cached release data is limited to public tag, date, URL, asset name, public download URL and size fields.
- No external data is inserted with `innerHTML`.
- Dynamic GitHub text remains rendered by the existing safe DOM code.
- No authentication, server, room, MLS, Pulse or download URL mapping is modified.
- `/advanced/` generation and content are not changed.

## Validation evidence

The branch validation path checks:

- JavaScript syntax for release fallback, resilience, `app.js`, `site-fixes.js` and `site-polish.js`;
- generated fallback freshness against every SemVer heading in `CHANGELOG.md`;
- regression coverage for timeout, fallback response and composition order;
- existing website and restored-UX validators;
- required motion tokens, menu lifecycle, `aria-current`, `aria-busy`, offscreen suspension and reduced-motion markers;
- deterministic JavaScript composition order: release fallback → request resilience → base → compatibility fixes → polish;
- published `build.json` marker.

## Real limitations

- When neither GitHub nor a prior browser cache is available, fallback entries contain no binary asset inventory. Platform buttons therefore open the selected GitHub Release page, while source links use the selected tag archive.
- Automated static validation cannot replace physical-device acceptance for iOS Safari, GPU-specific rendering or real 200% zoom interaction.
- Lighthouse and DevTools before/after traces require a browser runner and are not generated by the repository-only connector environment.
- The change intentionally avoids WebGL; visual depth is bounded by the existing Canvas 2D and CSS rendering model.
