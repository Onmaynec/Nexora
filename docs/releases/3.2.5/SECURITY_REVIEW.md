# Nexora 3.2.5 — Security Review

## Сохранённые границы

- ни один путь восстановления MLS не отправляет plaintext или приватные ключи на сервер;
- fast path применяется только к локально сохранённому проверенному MLS state и остаётся fail-closed при серверном отказе;
- запрос Welcome требует существующую session/Origin/CSRF/доступ к диалогу/verified device и серверный rate limit;
- консоль сервера остаётся фиксированным registry без shell, eval и произвольного доступа к файловой системе;
- локальная unsigned-сборка не меняет официальный release workflow: production updater assets публикуются только при наличии обоих Authenticode secrets;
- URL подробностей обновления строится только для текущей SemVer-версии и открывает официальный репозиторий `Onmaynec/Nexora`.

## Исправленный дефект SQLite

Pulse Sandbox создавал объект `billingLinks` с полем `localUserId`, тогда как persistence contract таблицы `billing_links` требовал `userId`. В результате SQLite получал `undefined` во втором параметре. 3.2.5 использует каноническое поле `userId` и нормализует старые snapshots без ручной миграции.

## Остаточные ограничения

- устройство, которое уже является MLS leaf, но физически потеряло локальный state, всё ещё требует безопасной повторной регистрации устройства; автоматическое восстановление не должно подменять identity или ослаблять forward secrecy;
- визуальное и installed-update тестирование подписанных NSIS-артефактов требует Windows-машину и действующий сертификат издателя.
