# Contributing to Nexora

Спасибо за вклад в Nexora. Этот документ описывает требования к изменениям, Pull Request и публичному взаимодействию в репозитории.

Участвуя в проекте, вы соглашаетесь соблюдать [Code of Conduct](CODE_OF_CONDUCT.md).

## Перед началом

- Для вопросов по установке и эксплуатации используйте [SUPPORT.md](SUPPORT.md).
- Для воспроизводимой ошибки создайте [Bug report](https://github.com/Onmaynec/Nexora/issues/new?template=bug_report.yml).
- Для улучшения продукта создайте [Feature request](https://github.com/Onmaynec/Nexora/issues/new?template=feature_request.yml).
- Уязвимости и секреты нельзя публиковать в Issue или Pull Request. Следуйте [SECURITY.md](SECURITY.md).

Перед крупным изменением сначала согласуйте границы через Issue. Это снижает риск реализации функции, которая не соответствует архитектуре или текущей дорожной карте.

## Локальная подготовка

Требуются Node.js `22.16+` и npm.

```bash
git clone https://github.com/Onmaynec/Nexora.git
cd Nexora
npm ci
npm run check
npm test
npm run audit:security
```

Не добавляйте native SQLite packages: Windows-сборка использует `node:sqlite` и не должна требовать `node-gyp`.

## Принципы изменений

- сохраняйте существующую архитектуру и переиспользуйте текущие сервисы, модели, компоненты и утилиты;
- исправляйте первопричину, а не скрывайте симптом;
- выполняйте авторизацию, проверку ролей, ограничений комнаты и входных данных на сервере;
- не считайте скрытый элемент интерфейса механизмом безопасности;
- изменения схемы оформляйте миграцией, проверкой совместимости и rollback-планом, если он поддерживается;
- сохраняйте совместимость API внутри major `3` либо документируйте migration и диапазон совместимых клиентов;
- изменяющие REST-запросы должны сохранять session, Origin и CSRF checks;
- production entitlement Plus/Pulse нельзя подтверждать локальным флагом;
- базовое общение и доступ пользователя к собственным данным не должны становиться платными;
- не добавляйте зависимости без обоснованной необходимости;
- не оставляйте TODO, заглушки, фиктивные данные, пустые обработчики или неиспользуемый код.

## Безопасность и приватность

Не коммитьте:

- `.env` и production credentials;
- SQLite-базы, backups и пользовательские вложения;
- CA/private keys, PFX/P12 и signing secrets;
- session cookies, API tokens, invite codes и Pulse credentials;
- реальные пользовательские данные в тестах, скриншотах или логах.

Для диагностических материалов используйте минимальный воспроизводимый пример и удаляйте чувствительные значения.

## Ветки и коммиты

Создавайте отдельную ветку от актуального `main`. Рекомендуемые префиксы:

- `feat/` — новая функциональность;
- `fix/` — исправление;
- `docs/` — документация и community files;
- `test/` — тесты;
- `chore/` — обслуживание без изменения поведения.

Commit subject должен быть коротким и в imperative style, например:

```text
fix: keep reaction picker interactive
```

Не смешивайте в одном Pull Request несвязанные изменения.

## Проверки

Минимальный набор перед Pull Request:

```bash
npm run check
npm test
npm run audit:security
```

Для релизно-чувствительных изменений также выполните:

```bash
npm run release:check
```

Дополнительно проверьте затронутый контур:

- Windows installers — `npm run dist:windows`;
- Android — `gradle -p android :app:assembleDebug`;
- долговременная целостность — `npm run test:soak`;
- UI — узкое окно, оба края экрана, keyboard navigation и details drawer.

## Pull Request

Pull Request должен содержать:

1. проблему и выбранное решение;
2. затронутые компоненты;
3. влияние на schema, API, security и compatibility;
4. добавленные или обновлённые тесты;
5. фактические результаты команд проверки;
6. migration/rollback notes при необходимости;
7. скриншоты или запись для визуальных изменений;
8. обновление документации и changelog для пользовательского поведения.

Ревью может быть приостановлено, если PR не воспроизводим, включает секреты, обходит серверные проверки или содержит несвязанный массовый рефакторинг.

## Документация

Документация должна описывать реальное поведение текущего кода. Не заявляйте функцию как готовую, если она не реализована и не проверена. Для Markdown используйте относительные ссылки на файлы репозитория и проверяйте, что они открываются из default branch.

## Лицензирование вклада

Отправляя изменение, вы подтверждаете, что имеете право предоставить его проекту и соглашаетесь распространять вклад на условиях [MIT License](LICENSE).
