# Руководство администратора Nexora 3.0.0

## 1. Назначение и границы

Nexora Server предназначен для Windows 10/11 и работает через localhost, LAN, Radmin VPN либо публичный HTTPS-домен. Для internet deployment используйте reverse proxy, действительный сертификат, ограниченный firewall и явные allowed origins; не публикуйте локальный порт простым port forwarding. Участник сети не считается автоматически доверенным: Server проверяет сессию, роль и доступ к каждому ресурсу.

## 2. Развёртывание

1. Установите Radmin VPN и подключите серверный ПК к нужной сети.
2. Установите подписанный `Nexora-Server-Setup-3.0.0.exe` либо выполните `npm ci && npm start` из проверенного source-релиза.
3. Разрешите входящий TCP 3443 только для частного профиля/Radmin-интерфейса. TCP 3080 нужен только для локального redirect на HTTPS.
4. Запустите Server и проверьте страницу «Обзор»:
   - статус «Сервер работает»;
   - полный адрес с отметкой RADMIN VPN;
   - уникальный Server ID;
   - SHA-256 сертификата;
   - SQLite integrity = ok.
5. Зарегистрируйте первый аккаунт — он станет администратором сервера.

Рабочие данные находятся в `userData\server-data`. Точный путь открывает кнопка «Открыть данные».

## 3. Client, браузер и сертификаты

Передавайте тестеру полный адрес `https://26.x.x.x:3443`, Server ID и SHA-256 по доверенному каналу.

Nexora Client показывает карточку нового сертификата. Пользователь сверяет отпечаток и нажимает «Доверять и подключиться». Client закрепляет PEM SHA-256 за Server ID; системная установка CA для `.exe` не требуется. Изменившийся сертификат всегда требует повторного подтверждения.

Для Edge/Chrome/PWA и Android при локальном CA экспортируйте `.crt` и установите его в доверенные сертификаты ОС, затем снова сравните отпечаток. Не предлагайте обходить предупреждение браузера или Android.

Если после подключения Radmin VPN адрес изменился, полностью перезапустите Server: сертификат должен содержать текущий IP в SAN.

## 4. Пользователи и сессии

В разделе «Пользователи» администратор может отключить аккаунт или выдать временный пароль. Сброс завершает активные сессии, а следующий вход требует обязательной смены пароля.

Пользователь самостоятельно управляет display name, bio/status, аватаром, паролем, TOTP/recovery codes, уведомлениями/quiet hours и активными сессиями. Блокировка запрещает личные сообщения и новые заявки; контакт можно удалить без удаления истории.

Настройте минимальную длину/классы пароля, число неверных попыток и время блокировки. Журнал входа хранит IP, результат и причину отказа.

## 5. Комнаты и модерация

Владелец назначает/снимает модераторов и передаёт владение. Владелец, модератор и server admin в пределах полномочий могут:

- удалить или забанить участника;
- просматривать room ban list;
- включить read-only/slow mode;
- отключить файлы или голосовые;
- переименовать комнату;
- рассматривать заявки;
- обновить/отозвать код, задать expiry и usage limit;
- выпускать несколько приглашений, создавать custom roles/categories;
- разбирать жалобы/апелляции и назначать временные ограничения;
- включить pre-approval сообщений и announcement mode;
- просматривать журнал действий.

Вступление и выход создают системные сообщения. Перед передачей владения проверьте, что новый владелец уже состоит в комнате.

## 6. Хранилище и резервные копии

Server использует SQLite schema 6, WAL и `synchronous=FULL`. Перед повышением schema создаётся backup; на старте выполняются миграции, `PRAGMA integrity_check`, cleanup expired sessions/uploads/files и поиск потерянных вложений.

В «Хранилище» задайте общую квоту и срок хранения файлов. До загрузки Client запрашивает доступную ёмкость. Значение retention `0` означает бессрочное хранение.

Для ручной защищённой копии задайте пароль не короче 10 символов. SQLite и вложения шифруются AES-256-GCM; ключ выводится через scrypt. Пароль нигде не сохраняется и не восстанавливается.

При restore Server сначала создаёт `pre-restore` копию, атомарно заменяет данные и повторно проверяет целостность. Храните хотя бы одну проверенную копию вне серверного компьютера. Не копируйте запущенный `nexora.sqlite` вручную.

## 7. Nexora Plus / Pulse

По умолчанию Pulse выключен. Режим задаётся до запуска Server:

```powershell
$env:NEXORA_PULSE_MODE = "disabled"   # либо sandbox / production
```

`sandbox` предназначен только для разработки: он активирует Plus и тестовые импульсы без денег. Не используйте его как production billing.

Для production нужны:

```powershell
$env:NEXORA_PULSE_MODE = "production"
$env:NEXORA_PULSE_CLOUD_URL = "https://billing.example.com"
$env:NEXORA_PULSE_API_KEY = "server-scoped-secret"
$env:NEXORA_PULSE_PUBLIC_KEY = "-----BEGIN PUBLIC KEY-----..."
```

Cloud обязан возвращать Ed25519-подписанные envelope. Server проверяет подпись, срок, Server ID/User ID и не выполняет денежную операцию при offline/invalid response. API key и приватный signing key нельзя хранить в репозитории.

До реальных продаж владелец должен подключить платёжного провайдера, обработку webhook/refund, налоги, оферту и политику конфиденциальности. Полный контракт: [docs/PULSE.md](docs/PULSE.md).

## 8. Обновления

Упакованный Client использует стабильные Releases публичного `Onmaynec/Nexora`: проверяет новую версию, автоматически загружает её и ставит при выходе. Только стабильный Release с подписанным installer, blockmap и `latest.yml` участвует в автообновлении; Source/PWA prerelease без Windows-подписи updater игнорирует.

Server обновляется только по решению администратора через HTTPS generic feed. До обновления создайте резервную копию. Server API v3 принимает Client major 2–3; несовместимая версия получает HTTP 426.

## 9. Боты и webhooks

Создавайте bot account только в нужной комнате и выдавайте минимальные scopes. Token показывается один раз и хранится Server только как hash; при утечке немедленно отзовите его. Webhook принимает только публичный HTTPS endpoint, подписывает payload HMAC и блокирует private/link-local адреса. Проверяйте `X-Nexora-Signature-256`, event ID и idempotency на принимающей стороне.

## 10. Журналы и авария

При сбое сохраните:

- `nexora-server.log` и при необходимости `nexora-client.log`;
- версии Client/Server и Windows build;
- Server ID, время и последние действия;
- состояние Radmin VPN и точный код ошибки без паролей/tokens/invite codes.

После отключения питания запустите Server и проверьте integrity. Если проверка не проходит, остановите Server и восстановите последнюю подтверждённую копию.

Admin API `/api/admin/metrics` показывает состояние памяти, диска, event backlog и базы; `/api/admin/runtime` включает аварийный read-only и выбирает Stable/Preview channel. Read-only не заменяет backup.

## 11. Выпуск Windows-релиза

На чистой Windows 10 и 11:

```bat
npm ci
npm run release:check
npm run audit:security
set NEXORA_SOAK_MINUTES=60
npm run test:soak
npm run release:windows
```

Проверьте Authenticode обоих `.exe`, чистую установку/удаление, upgrade с 2.0.0, сохранность SQLite и автообновление Client с предыдущего стабильного GitHub Release. Без signing secrets разрешён только Source/PWA prerelease. Подробный список: [docs/RELEASE_CHECKLIST.md](docs/RELEASE_CHECKLIST.md).
